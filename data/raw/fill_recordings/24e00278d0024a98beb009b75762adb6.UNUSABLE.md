# Capture 24e00278d0024a98beb009b75762adb6 — UNUSABLE for fill_price

**Status:** `fill_price-unusable`
**Captured:** 2026-05-13 18:00 UTC → 2026-05-14 09:06 UTC (~15h, exited early with `LastTaskResult=1`)
**Rows:** 110 (107 filled retcode=10009, 3 rejected retcode=10018)

## Why this capture is unusable

The `scripts/record_fills.py` version that produced this parquet read the
fill price from `mt5.OrderSendResult.price` directly:

```python
fill_price = float(order_send_result.price)  # BUG — reads 0 for market deals
```

MT5's `OrderSendResult.price` is **0.0 for market orders in most cases**.
The actual fill price lives in the subsequent deal record, accessible via
`mt5.history_deals_get(...)`. Because the script never queried the deal
record, every successful row landed with `fill_price = 0.0`. The
downstream `slippage_observed_pips` column then computes `(0 - requested_price) / pip`,
which produces absurd values like:

```
retcode=10009 requested_price=1.35149 fill_price=0.0  slippage_observed_pips= +13514.9
retcode=10009 requested_price=1.17077 fill_price=0.0  slippage_observed_pips= -11707.7
```

Those are the bug's signature — slippage magnitudes equal to the
requested price scaled by `1/pip`.

## Salvageable columns (DO use)

The bug only corrupts `fill_price`, `slippage_observed_pips`, and
`broker_fill_time_utc` (which fell back to wallclock because
`result.time == 0` too). The following columns are unaffected and remain
useful for follow-up calibration work:

- `retcode` — broker retcode is read directly from `OrderSendResult.retcode`.
- `requested_price` — internal to the request dict we built.
- `spread_at_request_pips` — derived from the bid/ask tick captured BEFORE the send.
- `broker_latency_ms` — Python-side wallclock RTT.
- `symbol`, `order_type`, `side`, `volume_lots`, `run_id`,
  `request_time_utc`, `comment` — all internal/passthrough.

## Unusable columns (DO NOT use)

- `fill_price` — always 0.0 on retcode=10009 rows.
- `slippage_observed_pips` — always absurd because computed from the zero `fill_price`.
- `broker_fill_time_utc` — always equal to `after_send_utc` (the wallclock
  fallback) because `result.time` was also 0; NOT broker-authoritative.

## Hard guard

`src/propfarm/gates/gate_2b.py` will refuse to run on this parquet
because the sibling manifest (`24e00278d0024a98beb009b75762adb6.json`)
has `"status": "fill_price-unusable"`. Regression test:
`tests/gates/test_gate_2b.py::test_run_gate_2b_rejects_unusable_manifest_status`.

## Fix

`scripts/record_fills.py` was patched at commit `9dd9af6` (2026-05-14) to resolve the fill
price + time from the deal record via `mt5.history_deals_get(...)` and
pass them into `parse_fill_into_record` as keyword args
(`actual_fill_price`, `actual_fill_time_utc`). The helper falls back to
NaN with a `retcode_or_deal_failure:` comment prefix on soft-failure
(retcode=10009 but no deal record yet).

See the `scripts/record_fills.py` module docstring section
"OrderSendResult vs. deal record" for the full convention and the
per-field audit.

## Secondary issue

The script also exited with `LastTaskResult=1` at iteration 110/200,
roughly 9h short of the 24h schedule. Root cause unknown without VPS
access, but the same fix-up commit adds a per-iteration try/except in
`main()` that:

- Logs the exception to **`sys.stderr`** (stdout is buffered/lost on
  hidden Task Scheduler jobs) with the structured prefix
  `[record_fills:exception]`.
- Includes the iteration index, symbol, order_type, side, and the full
  traceback.
- Continues to the next scheduled iteration.

End-of-loop summary lists `scheduled / attempted / exceptions /
exception-type-counts` so the operator can audit a run after the fact.
